# E分销平台 PRD

## 1. 文档信息

- 文档名称：E分销平台产品需求说明
- 文档类型：原型版 PRD
- 适用范围：管理端、经销商端、分销商端
- 当前版本：V1.0
- 更新时间：2026-03-16

## 2. 背景

E分销平台用于承接平台、经销商、分销商、服务商之间的分销业务协同，覆盖分销商准入、购销协议、订单协同、库存与基础配置等核心业务环节。

当前项目为长期原型系统，目标不是对接真实后端，而是通过可交互原型明确业务链路、角色职责、状态流转和页面结构，为后续需求澄清、方案评审和研发实施提供依据。

## 3. 业务主线

本模块建议从“分销商注册”开始理解，而不是从协议页面开始理解。完整业务主线如下：

1. 分销商提交注册申请，进入平台分销商池。
2. 平台对分销商基础资料和腾讯电子签认证状态进行管理。
3. 分销商完成认证后，平台才可以为其发起购销协议。
4. 购销协议进入签约审批。
5. 审批通过后，经销商补充乙方信息与条款。
6. 经销商补充完成后，分销商填写合同信息并发起电子签。
7. 服务商完成最终签署后，协议生效完成。
8. 在正式电签协议生成前，可执行平台协议作废。

## 4. 产品目标

### 3.1 业务目标

1. 建立 E 分销业务统一操作入口，沉淀平台、经销商、分销商三方协同流程。
2. 明确分销商签约、订单、库存、配置等关键节点的业务职责和状态。
3. 支持购销协议在平台内完成发起、审批、补充、签署前流转及平台作废。
4. 为后续真实系统建设输出可验证的页面原型和规则基线。

### 3.2 原型目标

1. 菜单结构清晰，符合真实业务分工。
2. 列表页、详情页、审批页结构统一，便于扩展。
3. 所有关键操作有可见状态反馈和明确流转结果。
4. 当前未实现模块优先保留菜单，不用随意填充伪业务内容。

## 5. 角色与职责

### 4.1 管理端

定位：平台运营与业务管理角色。

核心职责：
1. 管理分销商档案和签约状态。
2. 发起购销协议并进入签约审批。
3. 管理签约台账，查看协议全生命周期。
4. 在腾讯电子签正式协议生成前执行平台协议作废。
5. 管理服务商、平台配置、价格、库存、订单审批等平台侧能力。

### 4.2 经销商端

定位：服务商/经销商协同角色。

核心职责：
1. 查看并补充购销协议信息。
2. 在规则允许时执行平台协议作废。
3. 查看平台订单、店铺、价格、费用、OffTake等业务数据。

### 4.3 分销商端

定位：终端签约与履约角色。

核心职责：
1. 填写合同主体与收货信息。
2. 发起电子签流程。
3. 查看订单、收货地址、库存等业务数据。

## 7. 生命周期流程

### 7.1 分销商注册与认证流程

#### 业务目标

在发起协议前，先完成分销商主体准入和认证状态确认，确保后续签约对象合法、信息完整。

#### 流程说明

1. 分销商提交注册资料。
2. 平台生成分销商记录，初始可处于未认证或认证中状态。
3. 平台在分销商列表维护分销商主体信息、经营范围、联系人、电签人等基础信息。
4. 腾讯电子签认证状态作为是否可发起协议的重要前置条件。
5. 仅认证通过的分销商允许发起购销协议。

#### 关键状态

分销商认证状态包括：

1. 未认证
2. 认证中
3. 已认证
4. 认证驳回

#### 关键规则

1. 未认证、认证中、认证驳回状态不可发起协议。
2. 已认证状态才显示发起协议能力。
3. 分销商列表承担“注册后平台管理台账”的作用，是整个 E 分销平台链路的起点。

### 7.2 分销商签约主流程

1. 管理端在分销商列表为分销商发起购销协议。
2. 协议进入签约审批。
3. 管理端审批通过后，协议进入待服务商补充。
4. 经销商端补充乙方信息与条款信息后，协议进入待分销商签署。
5. 分销商端填写合同主体和收货信息并发起电子签后，协议进入待服务商签署。
6. 经销商端完成签署后，协议进入已签署完成。

### 7.3 平台协议作废流程

平台协议作废是平台内流程终止，不等同于腾讯电子签正式协议作废。

允许作废状态：
1. 待服务商补充
2. 待分销商签署

不允许作废状态：
1. 待签约审批
2. 待服务商签署
3. 已签署完成
4. 审批驳回
5. 已作废

作废规则：
1. 作废必须填写原因。
2. 作废后状态变更为已作废。
3. 作废后当前协议流程终止，不允许继续补充或签署。
4. 如需继续合作，需要重新发起协议。
5. 一旦已发起腾讯电子签生成正式协议，不再适用平台协议作废。

## 8. 协议状态设计

购销协议状态包括：

1. 待签约审批
2. 待服务商补充
3. 待分销商签署
4. 待服务商签署
5. 已签署完成
6. 已作废
7. 审批驳回

状态说明：

1. 待签约审批：等待平台审批，当前不允许作废，只允许通过或驳回。
2. 待服务商补充：审批通过后，经销商补充协议信息，可作废。
3. 待分销商签署：等待分销商填写并发起电子签，可作废。
4. 待服务商签署：分销商已发起电子签，平台协议作废关闭。
5. 已签署完成：协议完成。
6. 已作废：平台内协议流程已终止。
7. 审批驳回：审批阶段结束，不进入后续签约流程。

## 9. 页面需求

### 9.1 管理端-分销商列表

页面目标：
1. 承接注册后的分销商台账管理。
2. 查看分销商认证状态和当前签约状态。
3. 作为协议发起入口。
4. 支持重新发起协议。

页面结构：
1. 筛选卡片
2. 表格卡片
3. 右侧操作列

核心字段：
1. 分销商名称
2. 分销商编码
3. 企业主体类型
4. 社会信用代码
5. 法人
6. 联系方式
7. 电签人
8. 业务负责人
9. 经营产品范围
10. 认证状态
11. 签约状态
12. 注册时间

核心操作：
1. 发起协议
2. 重新发起协议
3. 查看已关联服务商

规则：
1. 仅已认证分销商可发起协议。
2. 已作废、审批驳回后允许重新发起。
3. 已签署完成或存在有效签约中记录时不允许重复发起。
4. 该页面是“从注册开始”的第一业务页面。

### 9.2 管理端-签约审批

页面目标：
1. 处理待签约审批协议。
2. 查看本人已审批记录。

核心操作：
1. 审批通过
2. 审批驳回

规则：
1. 审批通过后进入待服务商补充。
2. 审批驳回后流程终止。
3. 该页面不承接平台协议作废主入口。

### 9.3 管理端-签约列表

页面目标：
1. 作为协议全生命周期管理台账。
2. 统一查看协议状态和节点。
3. 统一执行平台协议作废。

页面结构：
1. 筛选卡片
2. 表格卡片
3. 详情页

核心字段：
1. 协议编号
2. 申请编号
3. 分销商名称
4. 分销商编码
5. 服务商
6. 当前节点
7. 当前状态
8. 发起时间
9. 审批时间
10. 作废时间

核心操作：
1. 查看详情
2. 作废协议

交互要求：
1. 作废采用按钮加弹窗方式。
2. 弹窗内必须输入作废原因。
3. 作废成功后列表和详情状态同步更新。

### 9.4 经销商端-购销协议列表

页面目标：
1. 查看当前购销协议台账。
2. 补充协议信息。
3. 在允许阶段执行平台协议作废。

核心字段：
1. 协议编号
2. 分销商名称
3. 服务商
4. 创建时间
5. 当前状态

核心操作：
1. 补充协议
2. 查看详情
3. 作废协议

规则：
1. 待服务商补充时可补充协议。
2. 待服务商补充、待分销商签署时可作废。
3. 已作废状态下只可查看，不可继续操作。

### 9.5 经销商端-购销协议详情

页面目标：
1. 查看协议主体信息。
2. 补充乙方信息与条款信息。
3. 查看作废信息。
4. 通过按钮弹窗执行作废。

条款区交互要求：
1. 条款项固定显示，不因表单编辑消失。
2. 默认规则中可编辑部分拆为独立输入值。
3. 百分号作为固定文案展示，不可编辑。
4. 非变量文案只读，不可编辑。

### 9.6 分销商端-合同详情

页面目标：
1. 填写甲方及收货信息。
2. 发起电子签。
3. 查看服务商已补充的条款内容。

规则：
1. 待分销商签署时允许填写并发起电子签。
2. 已作废状态只展示作废信息，不允许继续签署。

## 10. 关键业务规则

### 10.1 注册与认证

1. 分销商注册后进入平台管理视角。
2. 分销商是否可进入签约流程，取决于认证状态是否为已认证。
3. 认证状态是签约的前置门槛，不是协议状态的一部分。

### 10.2 发起协议

1. 单次发起仅选择一个服务商。
2. 发起后生成申请编号。
3. 审批通过后生成协议编号。

### 10.3 条款信息

1. 条款标题固定由模板提供。
2. 可变数值以单独输入控件维护。
3. 固定描述文案不可编辑。
4. 提交时系统将输入值拼装为规则文本进行保存。

### 10.4 平台协议作废

1. 列表页与详情页入口统一为按钮加弹窗。
2. 作废原因必填。
3. 作废后需记录作废时间、作废人、作废原因、作废来源。
4. 作废记录需在流转记录中可见。

### 10.5 重新发起协议

1. 已作废不算有效签约中。
2. 已作废后允许重新发起协议。
3. 已签署完成不允许直接重新发起。

## 11. 非功能要求

1. 当前阶段不接入真实后端接口。
2. 所有数据通过本地 mock service 驱动。
3. mock 数据与页面逻辑分离管理。
4. 菜单保持单一来源，页面注册保持在 pageRegistry。
5. 未实现页面使用占位或保留菜单，不填充不必要的伪内容。
